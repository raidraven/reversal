# 開発ログ(note商品化用の素材)

収益計画書([revenue-plan.md](./revenue-plan.md))Phase4-2「有料note(90日実況の全記録+使ったプロンプト全部)」のための素材ログ。
Claude Codeとのやり取りで「何を・なぜ・どう頼んだか」を開発のたびに追記していく。note執筆時はここから逆算して構成すればよい。

構成: 各エントリは `## 日付 タイトル` → 背景/課題 → 使ったプロンプト(要約 or ほぼ原文) → 何が起きたか → note向けのポイント(読者が学べること)。

## 2026-07-15 独自ドメイン(th-reversal.com)取得とVercel接続

**背景**: 収益計画タスク1-1。SEO上、独自ドメインなしでは不利になるため最優先で着手。

**使ったプロンプト・やり取りの流れ**:
1. 「リバーサルに寄せて考えて、SEO効果が高いものにしたい」→ ドメイン名の候補出し(EMD=完全一致ドメインのSEO効果は現在ほぼ無いという前提を正直に説明した上で、ブランド一貫性重視で候補を提示)
2. ユーザーが「th-reversal.com」を独自に取得
3. 「今のレンタルサーバー設定はどうなってる?」→ Vercelでホストされておりレンタルサーバーではないこと、DB(Postgres/Prisma)・NextAuth・Resend・Claude APIの構成を説明
4. 「th-reversal.comでドメイン発行しました」→ `vercel domains add`でVercelプロジェクトに追加(apex + www)
5. DNS未反映のトラブルシューティング — `nslookup`で権威DNSサーバーに直接問い合わせて確認しながら進行。原因は「ドメインのネームサーバー設定がレンタルサーバー(RSプラン)側を向いており、お名前.comの『ドメインDNS』ではなく『サーバーDNS』側の編集が必要だった」という、よくあるが分かりにくい罠
6. MXレコード(お名前メール)が既に存在していたため、ネームサーバー丸ごと切り替え(方法A)ではなく、既存ゾーン内でAレコード/CNAMEのみ書き換える方法B(低リスク)を選択
7. スクリーンショットを見ながら「ここをクリックしてください」を繰り返し、Aレコード書き換え→CNAME追加→`vercel domains verify`で最終確認
8. `NEXTAUTH_URL`環境変数の本番反映は、Claude Codeの自動実行ガードが「本番設定の未承認変更」としてブロック → ユーザーに明示確認を取ってから実行
9. 再デプロイ後、ブラウザで実際に`https://th-reversal.com`が表示されることを確認して完了

**つまずいたポイント(note向けに特に有用)**:
- ドメイン購入直後、DNSレコードを正しい場所で編集しないと何時間待っても反映されない(「ドメインDNS」と「サーバーDNS」が別物というGMO系レジストラ特有の罠)
- `nslookup <ドメイン> <権威サーバー>`で直接問い合わせると、リゾルバキャッシュを除外して「本当に反映されているか」を切り分けられる
- メール(MXレコード)が既にある場合、ネームサーバーを丸ごと切り替えるとメールが壊れるリスクがあるため、A/CNAMEだけを個別に書き換える方が安全
- Windows環境の`curl`は証明書失効チェック(schannel)でSSL自体は正常でもエラーになることがある(`--ssl-no-revoke`で切り分け)
- 本番環境変数の変更は、AI側の安全装置が自動実行をブロックする設計になっており、ユーザーの明示的な承認が必須だった

**noteのポイント**: 「ドメイン接続」は技術ブログでは1行で終わりがちだが、実際は非技術者がつまずく最大のポイントの一つ。スクリーンショットを見せながら一つ一つ確認していくプロセスそのものが、note読者(非エンジニアの副業実践者)向けの再現性が高い教材になる。「AIが本番変更を勝手に実行しない」という安全性の実例としても使える。

## 2026-07-15 GA4計測タグ導入・Search Console着手(タスク1-6)

**背景**: 独自ドメイン接続(1-1)完了により着手可能になったタスク1-6。「タスク1-6で」の一言で着手を指示された。

**やったこと**:
- GA4は「分からない」という回答だったため、まずGoogleアナリティクスとSearch Consoleの画面を実際に開いて確認するところから開始。既存のGA4アカウントに`nullプロパティ`という素性不明のプロパティが見つかったが、このサイト専用ではないと判断し、`th-reversal.com`向けに新規プロパティを作成
- 発行された測定ID(`G-LG9MM9FMBG`)を`next/script`でNext.jsに実装(`src/components/GoogleAnalytics.tsx`)。環境変数`NEXT_PUBLIC_GA_ID`が未設定なら何も出力しない設計にして、開発環境を汚染しないようにした
- Vercel本番・プレビュー両方に環境変数を設定し、再デプロイして計測タグの読み込みをブラウザで確認
- Search Console(ドメインプロパティ)の所有権確認でDNS TXTレコードによる認証を試行 → 反映確認で**大きなつまずき**が発生(下記)

**つまずいたポイント(重要)**: 「ドメインDNSが反映されているか」を`ns-rs1.gmoserver.jp`(お名前.comのレンタルサーバーパネルが案内するサーバー)に直接問い合わせて「反映済み」と判断していたが、実際にドメインへ委任されている権威サーバーは`01〜04.dnsv.jp`という別物だった。両者は同じGMOグループのインフラだが**同期にタイムラグがある**らしく、Aレコードは既に両方に反映済みだったのにTXTレコードだけ`dnsv.jp`側に反映されておらず、「反映済みと案内したのに実際は違った」という誤案内をしてしまった。`nslookup -type=NS <ドメイン> 8.8.8.8`で実際の委任先を確認すべきだった、という教訓。

**noteのポイント**: 「AIが自信満々に間違った情報を伝えることがある」という正直な失敗例。ここで重要なのは、失敗を隠さず「実際に権威サーバーを確認したら別物だった」と訂正した点。副業実況ものとして、AIとの協業の「うまくいかない時にどう修正するか」を見せる格好の題材になる。

## 2026-07-15 来賓数がカウントされない不具合を修正

**背景**: 「来賓者数がカウントされない」という一言の報告から調査を開始。

**原因**: 来賓の重複排除に使う匿名Cookie(`reversal_anon_id`)は、これまで「いいね」ボタンを押したときにしか発行されない実装になっていた(`src/app/api/posts/[id]/like/route.ts`等)。つまり閲覧だけして何もアクションしない大多数の来賓は、Cookieが一切発行されず、`landingStats.ts`側の重複排除キー(`visitorKey`)が`null`になって永久にカウント対象外になっていた。

**修正**: `src/middleware.ts`に、未ログイン・未発行の場合にCookieを発行するロジックを追加。単に発行するだけでなく、Next.jsの`NextResponse.next({ request: req })`パターンを使い、**同一リクエスト内**でServer Component(`page.tsx`)からも新しいCookieが読める状態にした(次回訪問を待たずにその場でカウントされる)。あわせて、元々`/home`等の認証必須ページにしか適用されていなかったmiddlewareのmatcherに`/`(LP)や`/articles`等の公開ページも追加。

**確認方法**: `curl`でCookie無しの初回アクセスをシミュレートし、`Set-Cookie`ヘッダーの発行と、レスポンスHTML内の来賓者数の数値が同一レスポンス内で増加していることを直接確認した(ブラウザのCookie画面に頼らず、生のHTTPレスポンスで検証する方法)。

**noteのポイント**: 「表示されている指標が実は正しく計測されていなかった」という、収益化を目指す上で地味だが致命的になりうるバグの実例。Next.jsのServer Component特有の制約(初回リクエストでCookieを発行できない)を、ミドルウェアの`request`差し替えパターンで解決した技術的に再現性のある解説ネタとしても使える。

## 2026-07-15 sitemap.xml・robots.txt実装(タスク1-9)

**背景**: 「sitemap実装かな」の一言で着手。1-8(書庫機能)完了で依存が解消していたタスク。

**やったこと**: Next.js App Routerの特殊ファイル(`sitemap.ts`/`robots.ts`)で自動生成する方式を採用。sitemapは公開記事をDBから動的に取得して自動的にURLリストへ含める(記事を書くたびに手動更新が不要)。robotsは認証必須ページ(`/home`, `/admin`など)を`Disallow`に指定。あわせてルートレイアウトに`metadataBase`を設定し、OGP・canonical URLの解決を正しくした。`SITE_URL`を一元管理する`src/lib/siteUrl.ts`を新設。

**noteのポイント**: 「記事を書くたびにsitemapを手動更新する必要がない」設計は地味だが継続運用のコストを下げる典型例。実況型で記事を増やしていく運用と相性が良い。Search Console側のドメイン所有権確認がDNS反映待ちで残っているため、送信は次回に持ち越し — 「技術は先に完成しても、外部サービス側の反映待ちで完了しない」という実況ものらしいリアルな進行状況。
